## 6. ChatGPT란 무엇인가

- 목적: ChatGPT의 이름과 작동 방식을 한 장으로 통합 설명한다.
- 핵심 내용:
  - ChatGPT는 대화형 생성 AI다.
  - Chat은 대화형 사용 방식을 의미한다.
  - Generative는 새로운 글, 표, 아이디어, 이미지 등을 생성한다는 의미다.
  - Pre-trained는 방대한 데이터를 미리 학습했다는 의미다.
  - Transformer는 문맥을 보고 다음에 올 가능성이 높은 표현을 예측하는 구조다.
  - 즉, ChatGPT는 사람처럼 대화하지만 실제로는 문맥을 바탕으로 다음 표현을 생성하는 모델이다.
- 화면 구성:
  - ChatGPT = 대화형 생성 AI
  - 하단에 Chat / Generative / Pre-trained / Transformer 간단 설명

---

## 7. LLM이란 무엇인가

- 목적: 대규모 언어 모델 개념을 쉽게 설명한다.
- 핵심 내용:
  - LLM은 Large Language Model의 약자다.
  - 많은 언어 데이터를 학습한 대규모 언어 모델이다.
  - 지금까지의 문맥을 보고 다음에 올 가능성이 높은 표현을 생성한다.
  - ChatGPT는 LLM을 사람들이 쉽게 대화형으로 사용할 수 있게 만든 서비스다.
- 화면 구성:
  - 데이터 학습 → 문맥 이해 → 답변 생성 흐름

## 8. ChatGPT는 사람처럼 쓰되, 더 명확하게 시켜야 한다

- 목적: 좋은 지시의 필요성을 설명한다.

  - 신입 직원에게 일을 맡기듯 구체적으로 설명해야 결과가 좋아진다.
- 화면 구성:

- 목적: 업무 활용 시 주의해야 할 보안 기준을 정리한다.
- 핵심 내용:

  - 개인 계정에서는 모델 개선 설정을 확인한다.

  - 최종 결과물은 사람이 검토한다.
- 화면 구성:
  - 보안 체크리스트

---

# 3부. 프롬프트 기초

## 20. 프롬프트란 무엇인가

- 목적: 프롬프트의 정의를 자연스럽게 설명한다.
- 핵심 내용:
  - 프롬프트는 ChatGPT에게 보내는 요청문이다.
  - 단순한 질문일 수도 있지만, 실무에서는 업무를 맡기는 지시문에 가깝다.
  - 좋은 프롬프트는 원하는 결과, 필요한 맥락, 출력 형식을 함께 알려준다.
- 화면 구성:
  - 단순 질문 vs 업무 요청 비교

---

## 21. 좋은 프롬프트는 무엇이 다른가

- 목적: 좋은 프롬프트의 특징을 한 장에 정리한다.
- 핵심 내용:
  - 나쁜 프롬프트:
    - “상세페이지 써줘”
    - “유튜브 제목 만들어줘”
    - “이거 정리해줘”
  - 좋은 프롬프트:
    - 누구를 대상으로 하는지 알려준다.
    - 어떤 결과물이 필요한지 말한다.
    - 참고할 자료나 배경을 준다.
    - 원하는 형식을 지정한다.
    - 피해야 할 표현과 검토 기준을 알려준다.
- 화면 구성:
  - Before / After 비교

---

## 22. 실무 프롬프트 기본 구조

- 목적: 프롬프트 구성 요소를 여러 장으로 쪼개지 않고 통합 설명한다.
- 핵심 내용:
  - 역할: 어떤 관점으로 답할지
  - 목표: 무엇을 만들지
  - 맥락: 누구에게, 어디에, 왜 쓰는지
  - 자료: 참고해야 할 원문이나 정보
  - 형식: 표, 대본, 목차, 체크리스트 등
  - 조건: 분량, 톤, 금지 표현, 주의사항
  - 검토: 빠진 내용, 오류, 개선점 확인
- 화면 구성:
  - 프롬프트 구성 요소 7개를 한 장에 배치

---

## 23. 기본 프롬프트 템플릿

- 목적: 바로 복사해 쓸 수 있는 기본 템플릿을 제공한다.
- 핵심 내용:

```text
너는 [역할]이다.

목표:
[무엇을 만들지 작성]

상황:
[대상, 사용 채널, 목적 작성]

참고 자료:
[필요한 자료 입력]

요청:
1. 핵심을 먼저 정리해줘.
2. 필요한 구조를 잡아줘.
3. 최종 결과물을 작성해줘.
4. 마지막에 개선점이나 주의점을 알려줘.

출력 형식:
[표 / 문단 / 대본 / 슬라이드 / 체크리스트]

조건:
[분량, 톤, 금지 표현, 난이도, 주의사항]
````

* 화면 구성:

  * 코드 박스 형태

---

## 24. 프롬프트를 개선하는 방법

* 목적: 후속 질문의 중요성을 설명한다.
* 핵심 내용:

  * 첫 답변이 마음에 들지 않아도 바로 버리지 않는다.
  * 더 짧게, 더 쉽게, 더 설득력 있게 수정시킬 수 있다.
  * 대상 고객 관점에서 다시 검토하게 할 수 있다.
  * 과장 표현이나 빠진 내용을 점검하게 할 수 있다.
* 화면 구성:

  * 초안 → 피드백 → 수정 → 최종화 흐름

---

# 4부. 프롬프트 작성 기법

## 25. Zero-shot Prompting

* 목적: 예시 없이 바로 요청하는 방식 설명
* 핵심 내용:

  * 예시 없이 바로 요청한다.
  * 빠르게 초안을 만들 때 좋다.
  * 결과가 일반적일 수 있다.
* 예시:

```text
부동산 초보자를 위한 유튜브 영상 제목을 10개 작성해줘.
```

* 화면 구성:

  * 간단한 요청 예시

---

## 26. One-shot / Few-shot Prompting

* 목적: 예시를 제공하는 방식 설명
* 핵심 내용:

  * One-shot은 예시 1개를 제공한다.
  * Few-shot은 예시 여러 개를 제공한다.
  * 기존 톤, 형식, 스타일을 맞출 때 유용하다.
* 예시:

```text
아래 예시와 같은 톤으로 제목을 작성해줘.

예시:
“부동산 공부, 이 순서로 안 하면 시간 낭비합니다”

주제:
초보자가 경매 공부를 시작하는 방법
```

* 화면 구성:

  * 예시 제공 전 / 후 비교

---

## 27. Role Prompting

* 목적: 역할을 부여하는 방식 설명
* 핵심 내용:

  * ChatGPT에게 어떤 전문가 관점으로 답할지 정해준다.
  * 같은 주제라도 역할에 따라 결과가 달라진다.
  * 마케터, 유튜브 PD, 교육 기획자, 고객 관점 등으로 바꿔볼 수 있다.
* 예시:

```text
너는 교육 상품 상세페이지 전문 마케터다.
아래 강의 상품의 상세페이지 구조를 설계해줘.
```

* 화면 구성:

  * 역할별 답변 방향 비교

---

## 28. Think step by step

* 목적: 단계별로 생각하게 하는 프롬프트 기법을 설명한다.
* 핵심 내용:

  * 복잡한 문제를 바로 답하게 하지 않고 단계적으로 생각하게 한다.
  * 핵심 문장은 “Think step by step” 또는 “단계별로 생각해줘”다.
  * 실무에서는 판단 기준을 먼저 세우고, 그 기준에 따라 결과를 만들게 하는 방식으로 활용한다.
* 예시:

```text
이 상세페이지의 문제점을 단계별로 생각해줘.
먼저 고객 관점의 판단 기준을 정리하고,
그 기준에 따라 문제점과 개선안을 제안해줘.
```

* 화면 구성:

  * 바로 답변 / 단계별 판단 비교

---

## 29. Self-consistency

* 목적: 여러 관점에서 답을 만들고 종합하는 방식 설명
* 핵심 내용:

  * 하나의 답만 받지 않고 여러 관점의 답을 비교한다.
  * 공통으로 중요한 내용을 뽑아 더 안정적인 결론을 만든다.
  * 기획, 마케팅, 개선안 도출에 유용하다.
* 예시:

```text
이 상세페이지를 3가지 관점에서 검토해줘.

1. 고객 설득 관점
2. 브랜드 신뢰 관점
3. 전환율 개선 관점

마지막에는 세 관점을 종합해 우선순위가 높은 개선안 5개를 정리해줘.
```

* 화면 구성:

  * 여러 관점 → 종합 결론

---

# 5부. 실제 업무 활용 프롬프트

## 30. 실무 적용 전에 먼저 정리할 것

* 목적: 바로 프롬프트를 쓰기 전에 업무 맥락을 정리하게 한다.
* 핵심 내용:

  * 이 결과물은 어디에 쓰는가?
  * 대상은 누구인가?
  * 고객이 가진 문제는 무엇인가?
  * 어떤 톤이 적합한가?
  * 반드시 포함해야 할 내용은 무엇인가?
  * 피해야 할 표현은 무엇인가?
  * 결과물은 어떤 형식이어야 하는가?
* 화면 구성:

  * 업무 맥락 정리 체크리스트

---

## 31. 실제 업무 유형 1: 콘텐츠 기획

* 목적: 영상, 블로그, 숏폼, 교육 콘텐츠 기획에 활용하는 방법 설명
* 핵심 내용:

  * 주제 발굴
  * 제목 후보
  * 오프닝 훅
  * 콘텐츠 구조
  * 핵심 메시지
  * 썸네일 문구
* 예시 프롬프트:

```text
너는 교육 콘텐츠 기획자다.

주제:
[주제 입력]

대상:
[대상 입력]

요청:
1. 이 주제에서 사람들이 궁금해할 질문 10개를 뽑아줘.
2. 콘텐츠 제목 후보를 10개 작성해줘.
3. 오프닝에서 사용할 후킹 문장 5개를 작성해줘.
4. 5분짜리 콘텐츠 구성안을 만들어줘.

조건:
과장된 표현은 피하고, 초보자도 이해하기 쉽게 작성해줘.
```

* 화면 구성:

  * 주제 → 제목 → 훅 → 구성안 흐름

---

## 32. 실제 업무 유형 2: 상세페이지·광고 문구

* 목적: 판매·홍보 문구 작성에 활용하는 방법 설명
* 핵심 내용:

  * 고객 문제 정리
  * 상품 장점 구조화
  * 상세페이지 섹션 구성
  * 광고 문구 생성
  * CTA 작성
  * 과장 표현 점검
* 예시 프롬프트:

```text
너는 교육 상품 마케터다.

상품:
[상품 설명]

대상:
[고객 설명]

요청:
1. 고객이 느끼는 문제를 정리해줘.
2. 상세페이지에 들어갈 섹션 구조를 제안해줘.
3. 각 섹션별 핵심 문구를 작성해줘.
4. 광고 문구 10개와 CTA 문구 5개를 작성해줘.
5. 과장되거나 위험한 표현이 있는지 점검해줘.

조건:
수익 보장, 과도한 불안 자극, 단정적인 표현은 피한다.
```

* 화면 구성:

  * 상세페이지 구조 + 광고 문구 예시

---

## 33. 실제 업무 유형 3: 강의·교육자료 제작

* 목적: 교육 콘텐츠와 강의 자료 제작에 활용하는 방법 설명
* 핵심 내용:

  * 강의 목차
  * 학습 목표
  * 차시 구성
  * 실습 과제
  * 퀴즈
  * 초보자용 설명
* 예시 프롬프트:

```text
너는 성인 교육 전문 커리큘럼 기획자다.

주제:
[강의 주제 입력]

대상:
[수강생 수준 입력]

요청:
1. 2시간 강의 목차를 설계해줘.
2. 각 파트별 학습 목표를 작성해줘.
3. 실습 과제 2개를 제안해줘.
4. 초보자가 어려워할 용어를 정리해줘.
5. 마지막에 강의 흐름이 자연스러운지 검토해줘.

출력:
표 형식
```

* 화면 구성:

  * 강의 목차 / 학습 목표 / 실습 과제 표

---

## 34. 실제 업무 유형 4: 고객 FAQ와 안내문

* 목적: 고객 응대와 안내 문서 작성에 활용하는 방법 설명
* 핵심 내용:

  * 자주 묻는 질문 정리
  * 답변 톤 통일
  * 신청 전 문의 대응
  * 수강 안내
  * 운영 공지
* 예시 프롬프트:

```text
너는 교육 서비스 CS 담당자다.

상황:
[고객 문의 상황 입력]

요청:
1. 고객이 궁금해할 질문 10개를 작성해줘.
2. 각 질문에 대한 답변을 친절하게 작성해줘.
3. 오해가 생길 수 있는 표현을 점검해줘.
4. 더 명확한 안내 문구로 다듬어줘.

조건:
친절하지만 과장하지 말고, 확정되지 않은 내용은 단정하지 말 것.
```

* 화면 구성:

  * 질문 / 답변 / 주의 표현 표

---

## 35. 실제 업무 유형 5: 자료 요약과 정리

* 목적: 긴 자료를 빠르게 이해하고 정리하는 방법 설명
* 핵심 내용:

  * 긴 문서 요약
  * 핵심 메시지 추출
  * 회의 내용 정리
  * 비교표 작성
  * 실행 항목 도출
* 예시 프롬프트:

```text
아래 자료를 업무에 바로 활용할 수 있게 정리해줘.

요청:
1. 핵심 내용을 5줄로 요약해줘.
2. 중요한 키워드를 뽑아줘.
3. 실행해야 할 일을 체크리스트로 정리해줘.
4. 추가로 확인해야 할 질문을 작성해줘.

출력:
요약 / 키워드 / 실행 항목 / 확인 질문으로 나눠서 작성
```

* 화면 구성:

  * 긴 자료 → 요약 → 실행 항목 도식

---

## 36. 실습 1: 내 업무 프롬프트 만들기

* 목적: 수강생이 자신의 업무에 맞는 프롬프트를 작성한다.
* 핵심 내용:

  * 업무 하나를 선택한다.
  * 대상, 목적, 자료, 형식, 조건을 정리한다.
  * 기본 템플릿에 맞춰 프롬프트를 작성한다.
* 실습 템플릿:

```text
업무:
대상:
목적:
참고 자료:
원하는 결과물:
출력 형식:
조건:
검토 기준:
```

* 화면 구성:

  * 빈 템플릿

---

## 37. 실습 2: 결과물 개선하기

* 목적: 첫 결과물을 그대로 쓰지 않고 개선하는 방법을 연습한다.
* 핵심 내용:

  * 첫 답변 검토
  * 부족한 점 찾기
  * 더 명확하게 수정
  * 실무 적용 가능성 점검
* 실습 프롬프트:

```text
방금 결과물을 검토해줘.

검토 기준:
1. 실무에서 바로 쓸 수 있는가
2. 대상에게 이해하기 쉬운가
3. 과장되거나 위험한 표현은 없는가
4. 빠진 내용은 없는가
5. 더 설득력 있게 만들 수 있는가

출력:
문제점 / 개선 방향 / 수정본
```

* 화면 구성:

  * 초안 → 검토 → 수정본

# 6부. 이미지 생성 AI 활용

## 38. 이미지 생성 AI란 무엇인가

* 목적: 이미지 생성 AI의 개념과 활용 범위를 설명한다.
* 핵심 내용:

  * 텍스트로 원하는 이미지를 설명하면 AI가 이미지를 만든다.
  * 기존 이미지를 바탕으로 수정할 수도 있다.
  * 상세페이지, 썸네일, 광고 배너, 강의 표지, 블로그 이미지, 카드뉴스 시안에 활용할 수 있다.
* 화면 구성:

# 50. Codex는 왜 소개할까?

## 슬라이드 제목

Codex는 개발자만 쓰는 도구일까?

## 화면 본문

Codex는 코드를 작성, 검토, 배포하는 데 도움을 주는 OpenAI의 코딩 에이전트입니다. 하지만 비개발자 교육에서는 코드를 직접 짜는 도구라기보다, **자동화 요구사항을 구조화하는 도구**로 소개하면 좋습니다. OpenAI도 Codex를 코드 작성·검토·출시를 돕는 AI 코딩 에이전트로 설명합니다. ([OpenAI Help Center][1])

비개발자 관점에서 Codex를 보는 방식

* 자동화하고 싶은 업무를 정리한다.
* 개발자에게 전달할 요구사항을 만든다.
* 브라우저나 시트 기반 업무 흐름을 설명한다.
* 예외 상황과 보안 조건을 정리한다.
* “가능한 자동화”와 “위험한 자동화”를 구분한다.

## 핵심 메시지

비개발자에게 중요한 것은 코딩이 아니라,
자동화할 업무를 명확하게 설명하는 능력입니다.

* Apps / Actions: 외부 도구와 연결하는 통로

## 핵심 메시지

GPTs는 “한 번 잘 만든 프롬프트”를

- 초보자도 이해할 수 있는 흐름으로 구성합니다.

출력 형식:

2. 전체 시간표
3. 모듈별 목표
4. 실습 과제

# 67. 자동화 워크플로우 실습 개요

## 슬라이드 제목

## 화면 본문

오늘 설계할 자동화 예시

### 챌린지 운영 자동화

Tally 제출
→ Google Sheets 기록
→ 운영자 알림
→ GPT로 제출 내용 요약
→ 댓글 초안 생성
→ 운영자가 확인 후 게시

## 자동화 단계

| 단계 | 도구            | 처리 내용          |
| -- | ------------- | -------------- |
| 입력 | Tally         | 챌린지 제출 폼       |
| 기록 | Google Sheets | 제출자, 링크, 내용 저장 |
| 알림 | Slack / 이메일   | 새 제출 알림        |
| 정리 | ChatGPT       | 제출 내용 요약       |
| 응답 | GPTs          | 댓글 초안 작성       |
| 검토 | 사람            | 최종 확인 후 게시     |

Tally는 Google Sheets, Slack, Notion, Make, Zapier 등과 연동할 수 있고, Tally 공식 문서에서도 다양한 통합 기능과 Google Sheets 연결, Slack 알림을 안내합니다. ([Tally Forms][8])

## 핵심 메시지

처음에는 완전 자동 댓글보다

6. 운영자가 확인 후 댓글 게시
7. 주간 리포트 자동 생성

## 주의할 점

네이버 공식 카페 API는 카페 가입과 게시글 쓰기 기능을 제공하며, 검색 API로 공개 카페글 검색도 가능합니다. 다만 댓글 자동 작성은 공식 공개 API 범위에서 제한적일 수 있으므로, 강의에서는 “자동 댓글 게시”보다 “댓글 초안 생성 + 운영자 승인” 방식으로 소개하는 것이 안전합니다. ([네이버 개발자][9])

## 핵심 메시지

커뮤니티 자동화는 속도보다 신뢰가 중요합니다.
댓글은 자동 작성보다 승인형 워크플로우가 적합합니다.
